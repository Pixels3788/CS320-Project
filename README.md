# CS320-Project

# How can I ensure that my code, program, or software is functional and secure?
Writing thorough unit tests is the most direct way I have found to ensure functionality. In this project, I tested every field constraint and every service operation, both the happy path and the error path so that any regression introduced by a future change would be caught immediately. For security and robustness specifically, the practice of testing null inputs and boundary values (such as strings exceeding maximum allowed length) ensures that validation logic actually holds up at the edges, not just in the middle of the valid range. Beyond testing, writing code with clear, enforced constraints at construction time, such as throwing an IllegalArgumentException before an invalid object can ever be created, prevents bad state from propagating through the system in the first place.

# How do I interpret user needs and incorporate them into a program?
The requirements for this project were expressed as explicit field constraints, such as maximum length, required formats, and rules about what values are and are not acceptable. I treated those constraints as the direct source of truth for both the production code and the tests. Each requirement became a question I had to answer with the code. This habit of translating requirements into concrete, testable conditions is something I want to carry into every project. It forces me to think carefully about what the user actually needs rather than what seems convenient to implement, and it produces software whose behavior is verifiable rather than assumed.

# How do I approach designing software?
 I approach design by starting with the boundaries of the system, what data comes in, what constraints it must satisfy, and what operations need to be supported before thinking about implementation details. For this project that meant designing the contact, task, and appointment classes around their validation rules first, then building the service layer on top. Separating the domain objects from the service layer was a deliberate design chouce that paid off in testing. Failures in the domain object tests pointed to validation logic, while failures in the service tests pointed to CRUD logic, making root causes easier to isolate. Going forward, I plan to continue thinking about the separation of concerns and testability as core design goals, not afterthoughts.
